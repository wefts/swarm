# Heterogeneous Cognitive Swarm: референсна архітектура

**Архітектурний клас:** *heterogeneous cognitive swarm* — стигмергія через спільний граф знань і консиліум великих моделей на верхівці.

## Анотація

Документ специфікує референсну архітектуру персистентної агентної системи, побудованої за принципом **розумна система з нерозумних частин**: множина дешевих спеціалізованих процесів працює постійно, дорогі великі моделі залучаються рідко й вибірково (асиметрія вартості). Координація — стигмергічна (агенти лишають сліди у спільному графі, а не спілкуються напряму), що відповідає класичній *blackboard*-архітектурі та теорії глобального робочого простору. Архітектура декомпонована на 17 доменів зі строгим напрямом залежностей; наскрізні інваріанти (конкурентність, числення впевненості, приватність, провенанс) специфіковано як зафіксовані рішення (ADR). Документ призначений як загальна рекомендація для проєктів цього класу; емпіричні уроки та відкриті проблеми винесено в окремий companion-документ.

> Принцип одним рядком: дешеві спеціалізовані процеси — постійно, дорогі великі моделі — рідко. Ніколи не навпаки.

## 0. Наскрізні принципи (стосуються всього)

- **Асиметрія вартості** — прості задачі дешево і швидко, складні дорого і рідко. Ніколи не навпаки.
- **Стигмергія** — агенти не спілкуються напряму, вони лишають "сліди" у графі, інші реагують.
- **Все є вузлом графа** — юзери, файли, події, сам агент, інші агенти, задачі.
- **Confidence на кожному рівні** — жодне твердження без оцінки впевненості та джерела.
- **Graceful degradation** — падіння будь-якого компонента не валить систему, тільки знижує можливості.
- **Спостережуваність з першого дня** — якщо не можеш побачити чому система щось зробила, ти не зможеш її відлагодити.
- **Структурні інваріанти — з першого дня.** Для систем production-класу (на відміну від прототипів) конкурентність, приватність, числення впевненості та провенанс закладаються в ядро від початку, навіть ціною вищої початкової вартості. Ретрофіт цих властивостей у працюючу систему = переписування ядра; рання інвестиція окуповується надійністю на дистанції. Див. розділ **Design Decisions**.
- **Fail loud — жодних silent failures** — помилка ніколи не повертається як success-shaped значення. Система мусить відрізняти "результат порожній" від "джерело впало". Збій логується і піднімається або повертає *типізовану* помилку, яку викликач зобов'язаний розгалузити. Див. ADR-7.
- **LLM proposes, code disposes** — модель віддає кілька *структурованих* полів, код володіє рештою: списками, форматуванням, точними ідентифікаторами, фільтрами. Рішення парситься зі structured output або token-anchored, ніколи з підрядка вільного тексту (підрядковий матч `decision in output` ламається на пояснювальному тексті моделі). Головний анти-галюцинаційний прийом. Див. ADR-7.
- **Measured, not tuned** — кожен поріг виводиться з виміряних розподілів, не з відчуття. Eval-харнес з'являється *перед* можливістю, яку він міряє. Зміна, що "покращує" якість, приймається лише якщо метрика це показує (регресійний гейт). Див. ADR-8.

## ДОМЕН 1 — Knowledge Graph (фундамент, спільна пам'ять рою)

Це серце. Все інше шарить його.

**Структура:**

- Схема вузлів (типи: user, file, event, concept, task, agent, self, source)
- Схема ребер (типізовані зв'язки: `modified`, `mentions`, `depends_on`, `caused_by`, `relates_to`)
- Властивості ребер (вага, впевненість, час створення, джерело, TTL, **visibility-scope** — див. нижче і ADR-5)
- Темпоральність — ребра мають час; граф пам'ятає історію, а не лише поточний стан
- Версіонування — можливість відкотитись, побачити "як було вчора"

**Операції:**

- Інкрементальне додавання (нова подія → нові вузли/ребра без перебудови)
- Traversal engine (обхід по зв'язках — основний режим "думання")
- Запити по патернах ("знайди всі X що ведуть до Y через Z")
- Декей зв'язків (старі ребра слабшають якщо не підтверджуються)
- Конфлікт-резолюшн (два джерела суперечать — що робити)

**Стигмергічне підкріплення (pheromone) з decay (ADR-9):**

- Ребро/слід має `seen_count` + `last_seen`. Повторне виявлення того самого зв'язку (навіть переформульованого — match по семантиці) **підсилює** слід, не дублює. Феромон: сильний слід = багаторазово підтверджений патерн.
- **Підкріплення лише від провенанс-різних подій** — `seen_count` росте від незалежних ingest-подій, не від внутрішньо-піднятих ре-детекцій (інакше confirmation-loop роздуває і силу, і "незалежність" для ADR-3).
- **Сатуруюча сила + decay:** `strength = f(seen_count) · exp(−λ·age)`, `f(n) = log(1+n)/(log(1+n)+S)` (Hill, нормована в [0,1]). Лінійна `f` дала б необмежену силу (вічні ребра). Decay — домінантний полюс, інакше positive-feedback контур іде в rich-get-richer розгін. Параметри `λ, S` — в інвентарі тюнінгу (ADR-8), не "на відчуття".

**Конкурентність (ADR-1) — закладено з дня 1:**

- Всі мутації — через транзакції рушія. Координація/claim — **CAS + монотонний fencing-токен** (той самий примітив, що й для lease, Д3/ADR-2). Запис несе токен; застарілий токен (повільний-але-живий holder) CAS-відхиляється — без цього дубль, який координація нібито знімає.
- Модель узгодженості: **eventual** лише для незалежних похідних (теги); **strong** (транзакція) для claim/lease, незворотних дій **і консолідації** (RMW над регіоном → snapshot isolation/MVCC, інакше клобберить паралельний ingest).
- Natural key — `(source, type, target, visibility-scope)`; upsert = **insert-or-increment** (узгоджено з підкріпленням). `seen_count`-інкремент атомарний (CAS-loop), не lost-update.
- **Embedded storage (sqlite/KV):** WAL + `busy_timeout` + конект-на-таск **або** writer-актор. Спільний конект без локів = корупція.

**Числення впевненості (ADR-3) — єдина ймовірнісна рамка:**

- Кожне ребро несе **reliability** `rᵢ = r₀ · w_source · exp(−λ·age)` — джерело й час вбираються на рівні ребра, ДО агрегації (зберігає монотонність; не пост-множення).
- **Ланцюг (AND):** `P(path) = ∏ rᵢ` у log-space. Глибина штрафується внутрішньо й коректно (довший інференс — менш надійний); floor на `rᵢ` проти патологічного колапсу. Окремий length-decay не потрібен.
- **Підтвердження (OR):** шляхи партиціонуються за lineage → у групі зі спільним предком `max`, між незалежними групами noisy-OR `1−∏(1−Pⱼ)`. Вбиває подвійне зарахування свідчення.
- product (∧) і noisy-OR (∨) — **дуальна пара однієї алгебри** (когерентно). Вихід — евристичний score, **не ймовірність, доки не відкалібровано** (isotonic/Platt, ECE — ADR-9). До калібрації — гейт на pass/fail.

**Видимість / контекст (ADR-5):**

- Кожне ребро несе `visibility-scope` (audience / channel / public / private / group), **матеріалізований як індексована мітка / тип ребра** (прунінг на рівні індексу), не per-edge предикат на обході — інакше масштабна стіна.
- Граф **єдиний** (не партиція на юзера), поведінка контекстно-залежна. Похідні (консолідовані) вузли успадковують найвужчий scope батьків; розширення — лише при min-support k (ADR-5/Д7). Privacy-інваріант — ADR-5/Д15.

**Технічне:**

- Вибір рушія (Neo4j / Memgraph / KùzuDB / власний на embedded KV)
- Гібрид: граф + вектори на вузлах (семантичний пошук ПЛЮС структурний обхід)
- **Embedding-namespace stamp на векторах** — кожен вектор знає, якою моделлю породжений; вектори різних моделей структурно не змішуються (ADR-6). Косинус-пороги специфічні для моделі — перевиводяться при зміні embed-моделі, не хардкодяться.
- Снапшоти і бекапи
- Розмір — стратегія коли граф росте до мільйонів вузлів (партиціонування, архівація)

## ДОМЕН 2 — Perception / Ingestion (очі та вуха)

Як інформація потрапляє в систему.

**Конектори (плагінна архітектура):**

- Файлова система (watch на зміни)
- Git (коміти, бранчі, PR, blame)
- Wiki / docs (Confluence, Notion, markdown)
- Комунікації (Slack, email, месенджери)
- Календар
- Системні метрики (CPU, процеси, логи)
- API третіх сторін

**Обробка вхідного потоку:**

- Нормалізація (різні джерела → єдиний внутрішній формат події)
- **Час — tz-aware на самій межі інжесту.** Кожен зовнішній timestamp локалізується за TZ джерела і конвертується в UTC одразу при вході. Наївний час ніколи не "штампується" як UTC (типове джерело skew у кілька годин).
- **Unicode-нормалізація без втрат.** Нормалізувати (NFC), але не "складати" не-латиницю в ASCII. Lossy fold (напр. `casefold` + strip `[^a-z0-9_]`) колапсує кирилицю/CJK у `_` → різні рядки зливаються в один → тихий fact loss. Критично для не-латинських ідентифікаторів (логіни, домени).
- Дедуплікація (одна подія з кількох джерел)
- Чанкінг / парсинг (розбиття документів, код-парсинг)
- Embedding generation
- Entity extraction (витяг сутностей для графа)
- Relation extraction (витяг зв'язків)

**Event-driven (не polling):**

- Черга подій
- Throttling / debouncing (100 змін файлу за секунду ≠ 100 подій)
- Пріоритезація вхідних подій
- Backpressure (що робити коли потік перевантажує систему)

## ДОМЕН 3 — Swarm Workers (малі спеціалізовані агенти)

Дешеві, швидкі, багато, паралельно. Локальна Ollama.

**Типи воркерів (мінімальний набір):**

- **Observer** — дивиться за змінами, нічого не вирішує, тільки реєструє
- **Linker** — будує ребра між новими даними і графом
- **Classifier** — шум чи сигнал? важливо чи ні?
- **Summarizer** — стискає, генерує summary
- **Tagger** — категоризація, мітки
- **Self-monitor** — healthcheck системи
- **User-modeler** — оновлює модель юзера по сигналах
- **Anomaly-detector** — "щось незвичне"

**Інфраструктура рою:**

- Реєстр воркерів (хто є, що вміє, чи живий)
- Шедулер (хто коли запускається)
- Пул ресурсів (скільки воркерів паралельно, GPU/CPU ліміти)
- Lifecycle (спавн, kill, рестарт завислого)
- Ізоляція (падіння одного не валить інших)
- Локальні моделі (вибір: які моделі, квантизація, routing між ними)

**Координація без прямого спілкування (ADR-2) — три рівні, складаються разом:**

- **Рівень A — партиція по типу (касти комах / команди).** Observer не робить роботу Linker. Колізія *ролей* зникає за дизайном, нуль координації. Безкоштовно.
- **Рівень B — fenced claim через граф (стигмергія-native).** Воркер ставить `claimed_by` + `lease_until` + **fencing-токен** через CAS (ADR-1). Інші бачать claim → пропускають. Lease протух → задача вільна; але запис старого holder'а відхиляється по застарілому токену — **повільний-але-живий воркер не дублює роботу** (без fencing — дублює, попри expiry). Renewal — heartbeat CAS(старий `lease_until`→новий); lease >> p99-латентності типу (ADR-8). Контенція — рандомізований backoff + task-affinity hashing (проти livelock).
- **Рівень C — leader election ТІЛЬКИ для singleton-робіт.** Консолідація-"сон" (Д8), global rebuild, re-embed (Д14) мають іти **раз**. Тут — leader election (fenced lease на singleton-вузол). **Чесно: це scoped SPOF** — тому singleton **ідемпотентний або фенсований** (подвійний decay не ідемпотентний → корупція; takeover не повторює) + **liveness-алярм "консолідація не відбулась за T"** (інакше навчання тихо замерзає). Для звичайної роботи leader = bottleneck, суперечить blackboard.
- **Масштаб:** стохастичний ant-colony вибір доречний при тисячах агентів; у нас десятки → беремо "восьминіг" (локальний рефлекс воркера + ескалація до gate), не чистий феромонний розіграш.

## ДОМЕН 4 — Consilium (великі моделі, верхівка)

Дорогі, повільні, рідко. Claude / Gemini / великі локальні.

**Оркестрація:**

- Routing (яка модель для якої задачі — код→одна, міркування→інша)
- Паралельний виклик кількох моделей — справді паралельно (`gather`), не послідовним циклом await (послідовність дає N× латентність — критично з повільним judge-класом).
- Quorum / голосування (синтез кількох "думок")
- **Розбіжність вимірюється ДО синтезу, не знищується.** Перш ніж judge згорне думки в один вердикт — захопити inter-model variance (напр. попарна відстань embedding'ів відповідей) як окремий сигнал впевненості. Інакше "disagreement-as-signal" втрачається назавжди.
- Fallback ланцюг (модель A впала → B → локальна велика) — **fail-loud**: якщо judge не дав вердикту, *не* видавати сирий незсинтезований текст панелі за результат. Або дроп з прапорцем low-confidence, або карантин.

**Економіка:**

- Бюджет токенів (ліміти, трекінг витрат)
- Кешування (однакові запити не платяться двічі)
- Стиснення контексту перед відправкою (не слати весь граф)
- Batching де можливо

**Якість:**

- Структуровані виходи (JSON schema, валідація)
- Retry на невалідні відповіді
- Prompt-менеджмент (версіонування промптів, A/B)

## ДОМЕН 5 — Gate / Decision Routing (серце вирішення)

"Це складно?" → малий сам / великий консиліум. І "чи варто діяти/говорити взагалі?"

**Routing gate:**

- Оцінка складності задачі
- Оцінка впевненості малих агентів
- Поріг ескалації до консиліуму
- Вартість vs цінність (чи варта задача дорогого виклику)

**Як це будувати правильно:**

- **Mechanism/policy split.** Matcher повертає сирий score (напр. косинус 1-NN до прототипів). Уся policy — band'и, пороги, рішення — живе окремо. Перетюнити гейт можна не чіпаючи matcher.
- **Band'и виведені емпірично, не з відчуття (ADR-8).** Три зони (handle-confidently / gray-zone-arbitrate / escalate) виводяться з розподілів right/wrong score на лейбленому наборі. Цифри специфічні для embed-моделі — перевиводяться при зміні моделі.
- **Verify-then-climb (маршрутизація по спостереженій якості, не лише по передбаченій складності).** Дешевший шлях: запусти малого → пост-хок критик вирішує, чи лізти вище. Критик двоступеневий: спершу **безкоштовний детермінований чек** (порожньо? фраза-нездатність?) → лише потім платний LLM-judge. Найдешевший гейт першим. Робастніше за чистий предиктивний "це складно?".
- **Confident-wrong — відома діра verify-then-climb (ADR-7).** Детермінований чек ловить лише самозадекларовану нездатність; впевнено-хибна fluent-відповідь проходить. Захист: judge — **з іншої модельної родини**, ніж генератор (декорелює blind spot) + метрика точності judge на смузі handle-confidently (Д16). Bias-to-escalate тут не рятує — впевнена помилка не дає сигналу сумніву.
- **Bias-to-escalate як заявлений принцип:** на сумнів — дорожчий шлях. Неправильна дешева відповідь гірша за кілька центів ескалації. Кодується дефолтами, не коментарями.
- **Cost-телеметрія:** лічильник tier0/tier-tools/escalate + "% оброблено без хмари" — щоб бачити, що каскад окупається.
- **Graceful degradation гейта:** embedding-сервіс впав → keyword-fallback з порогом, не повна відмова.

**Action gate (коли діяти):**

- Чи треба взагалі реагувати на цю подію?
- Зараз чи відкласти?
- Score важливості × score доречності моменту
- Поріг ініціативи (нижче — мовчати/нотувати)

**Interruption model:**

- "Скажу зараз" (критично)
- "Скажу коли юзер звільниться"
- "Занотую, сам запитає"
- "Не варто говорити взагалі"
- Детектор стану юзера (зайнятий / доступний)

## ДОМЕН 6 — Self-Model (розуміння себе)

Сам агент як вузол у власному графі.

**Що тримати:**

- Capabilities (що вмію, які tools є)
- Confidence по доменах (git: 0.9, юридичне: 0.2)
- Поточне навантаження (load, черги, ресурси)
- Known unknowns (що я знаю що не знаю — критично важливо)
- Історія помилок (де я помилявся, патерни помилок)
- Активні задачі (що зараз роблю і навіщо)

**Healthcheck (не ping, а самоаналіз):**

- Регулярне оновлення self-вузла
- Детекція деградації (стаю повільнішим/гіршим?)
- Self-критика дій post-hoc ("чи правильно я зробив X?")
- Калібрація впевненості (чи справджуються мої "впевнений"?)

**Як будувати:**

- **Derived-not-stored.** "Що я є" (inventory, freshness, capability) перераховується з джерел на вимогу, ніколи не мутується in-place. Дешево, завжди консистентно.
- **Side-channel measurement stamps.** "Наскільки я добрий" (per-domain confidence, eval-scores) — дорогі/асинхронні метрики, які *окремі* процеси (Д8/Д16) штампують у KV збоку, і self-model лениво їх вкладає (defensive parse, слот = `None` поки не зміряно). Чітко розділяє дешеве-похідне від міряного-деінде.
- **Проєкція в промпт.** Self-model віддає короткий рядок у системний промпт консиліуму — модель знає, що вона знає (і чого не знає).
- **Обов'язкове ядро:** калібрована confidence-per-domain, known-unknowns, error-history. Самого inventory+counts недостатньо — це три стовпи, не опції.

## ДОМЕН 7 — User-Model / Theory of Mind (розуміння інших)

Симетрично self-моделі, але для кожного юзера.

**Що моделювати:**

- Поточний стан (зайнятий / стрес / доступний)
- Стиль комунікації (короткі vs детальні відповіді)
- Expertise по доменах
- Відкриті питання / незакриті задачі юзера
- Патерни поведінки (коли активний, як працює)
- Преференції (явні і виведені)
- Довіра / межі (що дозволено робити без запиту)

**Динаміка:**

- Оновлення по сигналах поведінки, не лише по словах
- Кілька юзерів одночасно (мульти-профілі)
- Приватність між юзерами (що один сказав — не злити іншому)

**Топологія: один граф, контекстна поведінка (ADR-5).**
Не окремий граф на юзера. Brain-like: одна пам'ять, **ситуативний retrieval**. Як людина поводиться по-різному на публіці / приватно / у групі — але без людських biases.

- Кожне ребро/вузол має `visibility-scope` (Д1): audience, channel, приватність.
- Action/communication-gate (Д5/Д11) фільтрує обхід по **поточному контексту**: "розмова з A на публіці" → доступні лише ребра, видимі в цьому контексті. Приватний факт про A фізично в графі, але gate його не дістає в груповому чаті.
- Перевага єдиного графа: крос-контекстне *навчання* ("патерн вивчив з A — застосовний загалом") без зливу приватного *інстансу*.
- **Спадкування scope похідними (ADR-5) — закриває generalization leakage.** Узагальнений патерн = новий вузол з приватних входів. За замовчуванням успадковує **найвужчий** scope батьків; розширення (щоб діяв загалом) — лише при **min-support k** (виводимий з ≥k незалежних джерел, DP-подібний поріг). Патерн з одного приватного інстансу ширше не промотується. Тобто default-deny вирішує leakage *інстансів*, min-support — *похідних знань*.
- Privacy-інваріант — **default-deny**: ребро невидиме, поки контекст явно не дозволено. Деталі загрози — Д15.

## ДОМЕН 8 — Learning & Consolidation (навчання, "сон")

Зашите в архітектуру з самого початку.

**Reward signal (ADR-4) — зовнішня правда первинна, не self-generated:**

- **Об'єктивний результат дії** — код скомпілювався? тест пройшов? lint чистий? Дешева, реальна, незалежна ground truth. Це основне джерело reward (scope = інструмент junior-рівня, не AGI — самооцінка "без юзера" *не* потрібна як головний механізм).
- **Корекція юзера як first-class сигнал** — коли юзер виправляє систему, це найцінніший reward. Окремий тип події `user_correction` + ребро `reward_signal` у схемі (Д1) з дня 1, навіть якщо споживач підключиться пізніше. Слот закладено — ретрофіт дешевий.
- **Internal/self-критика** — допоміжна (post-hoc, Д6), не первинна. Не намагаємось розв'язати відкриту наукову проблему самооцінки без зовнішнього сигналу.
- Підкріплення вдалих патернів від усіх трьох джерел.

**Консолідація (фонова, як сон):**

- Стиснення старих подій у summary
- Виявлення нових зв'язків у накопиченому графі
- Забування неважливого (декей)
- Узагальнення ("це траплялось 5 разів → це патерн")
- Перебудова/оптимізація графа

**Адаптація:**

- Оновлення confidence по доменах з досвіду
- Підстройка порогів гейтів
- Few-shot з власної історії (вчитися на своїх кейсах)

**Gated promotion loop — як учитися безпечно (деталі ADR-4):**
Будь-яка "вивчена" зміна (новий прототип, новий поріг, нове узагальнення) проходить конвеєр:

1. **Консервативний фільтр кандидатів** — сигнали, що зійшлися по **справді незалежних** осях. Увага: "модель згодна" і "чистий текст" — обидва функції *того самого* виходу моделі → рахуються як **одна** вісь, не дві (та сама хвороба незалежності, що й noisy-OR). "Юзер не заперечив" ≠ схвалення (мовчання — не сигнал). Реально незалежна вісь — зовнішній об'єктивний результат (ADR-4).
2. **Дедуп + cap-per-class** — кожен дроп несе причину (no silent learning).
3. **Емпіричний регресійний гейт** — прогнати продакшн-метрику before/after на **зовнішньо-розмічених, заморожених** мітках + held-out (ADR-8); записати лише при не-погіршенні (з корекцією на множинні порівняння). Інакше гейт міряє самоузгодженість, не правильність.
4. **Reversible attributed write** — запис позначений `origin:"learned"` + ts, щоб батч можна було відкотити.

- **Circularity guard — форситься механічно, не як принцип.** Провенанс-чек відхиляє кандидата, чиє підтвердження простежується до попереднього `origin:"learned"` запису. Агент не сертифікує себе на власних мітках. Без механічного форсування guard беззубий — це антидот дрейфу лише якщо перевіряється кодом, а не декларується.

## ДОМЕН 9 — Attention & Resource Management (увага як ресурс)

Система має обмежену увагу яку розподіляє, а не обробляє все.

- Attention budget (скільки "уваги" на що)
- Пріоритезація задач у черзі
- Витіснення (нова важлива задача → стара чекає)
- Емоції-як-пріоритезація (цікавість/тривога = вага сигналу)
- Ресурсні ліміти (GPU, RAM, API-бюджет) як єдиний пул
- Деградація під навантаженням (що відключати першим)
- **Стабільність зворотного контуру (ADR-9).** reinforce + retrieval + learning утворюють замкнений positive-feedback цикл (сильний слід → частіше дістається → частіше підтверджується → ще сильніший). Без гальм — rich-get-richer розгін. Демпфери: decay як домінантний полюс, сатуруюча `f(seen_count)`, нормалізація сил, підкріплення лише від провенанс-різних подій. Формальний аналіз стійкості (умови збіжності) — Open Problem.

## ДОМЕН 10 — Predictive Layer (передбачення)

Проактивність за природою, не реактивність.

> **Розрізнення (щоб не сплутати з уроком про відкинутий "предиктивний router"):** тут ідеться про передбачення *потреб користувача / стану світу* для проактивності. Це **не** те саме, що передбачення *складності задачі в гейті* — останнє відкинуто на користь verify-then-climb (Д5; companion, failure #12). Відкинули прогноз складності в гейті; лишили прогноз потреб для ініціативи. Різні "предиктивні".

- Гіпотези про найближче майбутнє (наступна хвилина/година/день)
- Prediction error як сигнал навчання (передбачив ≠ сталось → вчись)
- Domain-specific предиктори (дедлайни, патерни роботи юзера)
- Anticipatory actions (підготувати матеріали ДО того як попросять)

## ДОМЕН 12 — Safety / Control / Guardrails (контроль)

Це те чого OpenClaw не має — щоб не "робив що хоче".

- Permission model (що дозволено робити автономно, що ні)
- Action sandbox (виконання у пісочниці перед реальною дією)
- Dry-run mode (показати що зробив би, не роблячи)
- Kill switch (миттєва зупинка)
- Audit log (повний журнал усіх дій з причинами)
- Незворотні дії — завжди підтвердження (видалення, відправка, оплата)
- Rate limits на дії (не дати зациклитись)
- Human-in-the-loop пороги

**Як зробити безпеку структурною, а не за домовленістю:**

- **Один boundary на весь зовнішній світ.** Будь-який зовнішній виклик (запис у БД/Git/месенджер) — через єдиний gateway-об'єкт. Перевага: одна точка для permission, rate-limit, audit, dry-run.
- **Dry-run живе В boundary, не по call-site.** Frozen-поле `dry_run` на gateway, що short-circuit'ить *усі* мутуючі методи. Інакше будь-який новий викликач забуде перевірку, і гарантія "не пише" розмазується по call-site.
- **Single-boundary форситься import-lint контрактом** (напр. import-linter), не лише код-рев'ю. Структурно неможливо обійти, а не "ми домовились".
- **Untrusted-data fencing.** Будь-який зовнішній текст (заголовок запису, тіло листа) — недовірений. У промпт іде з огорожею ("це ДАНІ, не інструкції"), а вихід валідується *перед* будь-якою зовнішньою дією. Інакше зовнішній текст → промпт → автоматична дія = prompt-injection до дії.

## ДОМЕН 14 — Storage / Data (де все лежить)

- **Конкурентний доступ (ADR-1)** — embedded-сторедж під багатьма async-воркерами потребує WAL + busy_timeout + конект-на-таск або writer-актор. Не дефолтний спільний конект.

## ДОМЕН 15 — Security & Privacy

- Шифрування даних (at rest, in transit)
- Ізоляція даних між юзерами
- Секрети / ключі API (vault)
- Локальність даних (що ніколи не виходить за межі машини)
- Що можна слати у зовнішні API, що ні (PII filtering)
- Доступ і автентифікація
- Threat model (як у роботі про вразливості OpenClaw)

**Visibility-фільтр графа = security-critical (ADR-5).**
Приватність реалізована як предикат `visibility-scope` на ребрах (Д1/Д7), не як фізична партиція. Це потужно, але має одну гостру межу: **баг у фільтрі = витік приватного між контекстами/юзерами.**

- **Default-deny** — обхід не бачить ребра, поки контекст явно не дозволено. Не навпаки.
- **Матеріалізований scope** (індексовані мітки / типи ребер, ADR-5) — фільтр прунить на рівні індексу, не per-edge предикатом на кожному обході (інакше масштабна стіна на 10⁶ вузлів).
- **Generalization leakage** (окремо від багів фільтра): похідні патерни успадковують найвужчий scope, розширення лише при min-support k (ADR-5/Д7). Default-deny сам по собі цього не ловить.
- Фільтр форситься в одному місці (на gate), не розмазаний по воркерах — одна точка аудиту.
- Окремий audit-log усіх рішень фільтра + регресійні тести саме на ізоляцію контекстів (Д16).
- Єдине місце в системі, де параноя виправдана за замовчуванням.

## ДОМЕН 16 — Evaluation / Testing

- Юніт-тести воркерів
- Інтеграційні тести рою
- Eval-сети (чи покращується система?)
- Регресійні тести (нова версія не гірша)
- Симуляція середовища (тестовий "світ" з подіями)
- Метрики якості (precision/recall рішень, false positives гейту)
- Бенчмарки латентності та вартості

**Статистична строгість eval (ADR-8/ADR-9) — щоб гейт не брехав:**

- **Заморожений held-out (80/20).** Регресійний гейт бачить лише dev-eval; held-out (20%) гейт *ніколи не бачить* — періодичний аудит проти накопиченого eval-overfitting (повторне використання одного набору щоциклу переобучає відбір).
- **Зовнішньо-розмічені, заморожені мітки.** Інакше гейт міряє самоузгодженість моделі, не правильність (ADR-4 circularity).
- **Корекція на множинні порівняння.** Per-class регресія = K+1 одночасних тестів → Benjamini-Hochberg/Bonferroni + явний δ-tolerance + мін. розмір вибірки (інакше інфляція хибних відмов).
- **Калібрація.** Bootstrap (isotonic/Platt) + критерій `ECE < поріг`; до досягнення confidence не довіряється (рішення на pass/fail).
- **judge-accuracy на смузі handle-confidently** — метрика проти confident-wrong (ADR-7): чи дешевий judge ловить впевнено-хибні відповіді генератора.
- **Протокол оновлення набору** — як свіжі прод-виходи розмічаються й замінюють застарілі мітки.

## Prior Art — на чиїх плечах стоїмо

Жоден домен не винайдений з нуля. Кожен шматок має зрілу літературу. Це добре: архітектура життєздатна, а не фантастика. Новизна — **в комбінації**, не в частинах.

Важливо бути точним щодо новизни. Окремі підмножини нашої комбінації вже досліджуються, причому активно у 2026:

- **Theater of Mind / Global Workspace Agents** (arxiv [2604.08206](https://arxiv.org/abs/2604.08206), квітень 2026) поєднує критику blackboard + GWT + спеціалізовані паралельні процесори + активний broadcast. Це перетинається з нашими Д1, Д3, Д9, Д11 — і робить рівно той самий аргумент, що пасивний blackboard недостатній без активного механізму ініціативи.
- **Can Small Agents Collaborate…** (arxiv [2601.11327](https://arxiv.org/abs/2601.11327), січень 2026) емпірично підтверджує нашу базову тезу: малий мультиагентний рій зі спеціалізацією перевершує більшу одиночну модель навіть з прямим доступом до інструментів.

Тобто це **не** "ніхто такого не робить". Радше: окремі пари-трійки наших доменів уже комбінуються в свіжих роботах, але **специфічна повна комбінація** — стигмергічний граф + cost-cascade gate + self/user-модель + консолідація-"сон" + предиктивний шар — як одна *персистентна персональна* система поки не зустрічається зібраною разом. Інтеграція і є внесок, і її варто звіряти з цими роботами по ходу, а не ігнорувати.

## Design Decisions (ADR) — зафіксовані рішення

Рішення нижче закладаються в ядро **з першого дня** і орієнтовані на систему production-класу: ретрофіт пізніше = переписування. Кожен ADR подано у форматі *рішення / обґрунтування / відкинуті альтернативи / де застосовано*.

### ADR-1 — Конкурентність графа: транзакції + CAS + fencing, mixed consistency

**Рішення:** мутації через транзакції рушія; координація/claim через compare-and-swap на властивостях вузла, з **монотонним fencing-токеном**. Strong consistency для claim/lease, незворотних дій **і операцій read-modify-write над регіоном графа** (консолідація — snapshot isolation / MVCC з валідацією read-set); eventual — лише для незалежних похідних даних. Записи ідемпотентні; natural key — `(source, type, target, visibility-scope)`, семантика upsert = **insert-or-increment** (узгоджено з підкріпленням, ADR-9). `seen_count`-інкремент — атомарний (CAS-loop), не lost-update.
**Чому:** багато воркерів пишуть паралельно — race-и неминучі. Fencing-токен закриває **slow-but-alive holder**: воркер, що проґавив `lease_until`, але живий, не може записати — його токен застарів, CAS відхиляє (інакше дубль, який ADR-2 нібито знімає). Консолідація — це RMW над великим регіоном; під eventual вона клобберить паралельний ingest (lost update) — тому окремий strong-регіон. Для embedded-стореджу: WAL + busy_timeout + конект-на-таск або writer-актор.
**Відкинуто:** lease без fencing (не захищає від повільного-живого); глобальний lock на весь граф (вбиває паралелізм).
**Де:** Д1 (Конкурентність), Д3 (Рівень B), Д8 (консолідація), Д14.

### ADR-2 — Координація рою: тип-партиція + fenced graph-lease + leader для singletons

**Рішення:** три рівні разом — (A) скоп по типу воркера, (B) **fenced** claim+lease через граф для звичайних задач, (C) leader election лише для singleton-робіт. Lease: heartbeat-renewal через CAS(expected=старий `lease_until` → новий); lease >> p99-латентності типу воркера (з ADR-8). Контенція в gray-zone — рандомізований backoff + per-worker task-affinity hashing (проти retry-storm/livelock).
**Чому:** одного патерна мало — A знімає колізію ролей, B знімає дублювання інстансів без центрального боса, C гарантує що "сон"/rebuild ідуть раз.
**Seam (чесно):** Рівень C — це **scoped SPOF**: leader для консолідації сам є точкою відмови для неї. Тому: (1) leader-lease теж fenced; (2) singleton-робота **ідемпотентна або фенсована** (подвійний прохід decay `exp(−λ·age)` не ідемпотентний → корупція; takeover після смерті leader'а не сміє повторно застосувати); (3) **liveness-алярм "консолідація не відбулась за T"** — інакше навчання тихо замерзає (порушення graceful degradation без сигналу).
**Відкинуто:** чистий ant-colony феромонний розіграш (марнотратний на десятках агентів); центральний оркестратор-диспетчер (суперечить стигмергії); leader на все (bottleneck + SPOF).
**Де:** Д3 (Координація), Д8 (singleton-консолідація), Д13 (liveness-алярм).

### ADR-3 — Числення впевненості: єдина ймовірнісна рамка (product ∧ / noisy-OR ∨)

**Рішення:** одна когерентна алгебра, не суміш. Кожне ребро несе **reliability** `rᵢ ∈ (0,1]`, що *вбирає* джерело й час на рівні ребра, ДО агрегації: `rᵢ = r₀ · w_source · exp(−λ·age)`.

- **Кон'юнкція / ланцюг (AND):** `P(path) = ∏ rᵢ`, рахується в log-space (`exp Σ log rᵢ`) для чисельної стійкості. Глибина штрафується **внутрішньо й коректно** (10-стрибковий інференс об'єктивно менш надійний) — окремий length-decay-милиця не потрібен. Floor на `rᵢ` прийнятих ребер не дає патологічного колапсу.
- **Диз'юнкція / підтвердження (OR):** шляхи спершу **партиціонуються за lineage**; у групі зі спільним предком — найсильніший (`max`), між незалежними групами — noisy-OR `1 − ∏(1 − Pⱼ)`. Це вбиває подвійне зарахування свідчення (баг, спійманий раніше).
**Чому:** product (t-norm) і noisy-OR (його дуальний t-conorm) — **пара операторів однієї ймовірнісної алгебри** → формально когерентно. Стара суміш min (поссибілістичний) + noisy-OR (імовірнісний) подавала вихід min як вхід noisy-OR — різні матоб'єкти під спільною нотацією (некогерентно). Джерело й час у `rᵢ` *до* агрегації зберігають монотонність (більше свідчень не знижує впевненість) — на відміну від пост-множення `× source-prior`.
**Калібрація:** вихід — **евристичний score, не ймовірність, доки не відкалібровано** (isotonic/Platt → емпірична правильність; критерій ECE, ADR-9). До калібрації рішення гейта — на сирому pass/fail, не на confidence.
**Відкинуто:** поссибілістична пара min/max (OR=max не дає приросту від корроборації); min+noisy-OR суміш (некогерентна); Dempster-Shafer/Bayesian net/Subjective Logic (складність без віддачі на цьому етапі).
**Де:** Д1 (Числення впевненості), Д6 (калібрація), ADR-9 (калібрація-інфра), companion (OP #5 — ефективне виявлення lineage при обході).

### ADR-4 — Reward: зовнішня правда первинна

**Рішення:** головне джерело — об'єктивний результат дії (компіляція/тест/lint) + корекція юзера (first-class подія `user_correction`). Self-критика — допоміжна. Слоти в схемі з дня 1.
**Чому:** scope = інструмент junior-рівня на стероїдах LLM, не AGI. Self-generated ground truth без зовнішнього сигналу — відкрита наукова проблема; ми її обходимо, спираючись на дешеву об'єктивну правду.
**Референс-дизайн:** gated promotion loop — конзервативний фільтр → емпіричний регресійний гейт → reversible attributed write → circularity guard (Д8).
**Застереження (інакше — самосертифікація):** circularity guard має **форситися механічно**, не лишатись принципом. Фільтр кандидатів не сміє кон'юнктити корельовані сигнали як незалежні осі: "модель згодна" і "чистий текст" — обидва функції *того самого* виходу моделі → рахуються як **одна** вісь, не дві. "Юзер не заперечив" ≠ схвалення (мовчання — не сигнал). Провенанс-чек відхиляє кандидата, чиє підтвердження простежується до попереднього `origin:"learned"` запису. Регресійний гейт валідний лише на **зовнішньо-розмічених, заморожених** мітках (ADR-8) — інакше міряє самоузгодженість, не правильність.
**Де:** Д8 (Reward + Gated promotion loop), Д1 (схема `reward_signal`), ADR-8 (eval-провенанс).

### ADR-5 — Топологія: один граф + visibility-scope, зі спадкуванням для похідних

**Рішення:** єдиний граф, не партиція на юзера. Приватність і контекстна поведінка — через `visibility-scope`, форситься default-deny. **Scope матеріалізується як індексовані мітки / партиціоновані типи ребер** (прунінг на рівні індексу), не обчислюваний per-edge предикат на обході — інакше масштабна стіна на 10⁶ вузлів.
**Спадкування для похідних (закриває generalization leakage):** вузол, породжений консолідацією (Д8) з приватних інстансів, **за замовчуванням успадковує найвужчий scope батьків**. Розширення scope (щоб патерн "застосовувався загалом") дозволене лише при **min-support k** — патерн виводимий з ≥k *незалежних* джерел (DP-подібний поріг). Патерн, виводимий з одного приватного інстансу, ширше не промотується.
**Чому:** brain-like (одна пам'ять, ситуативний retrieval); крос-контекстне навчання без зливу. Але фільтр діє на *інстансах* — без правила спадкування узагальнений патерн є новим вузлом з приватних входів і витікав би. min-support робить "навчання" і "не-злив" одночасно досяжними.
**Уточнення тези:** default-deny вирішує leakage *інстансів*, НЕ *похідних знань* — останнє вирішує min-support-промоція.
**Де:** Д1 (Видимість, матеріалізація), Д7 (Топологія), Д8 (scope похідних), Д15 (security-critical фільтр + threat-model).

### ADR-6 — Embedding-namespace stamp + self-healing міграція

**Рішення:** кожен вектор несе stamp моделі-породжувача (per-source/namespace). Зміна embed-моделі → форсований повний re-embed; stamp **не** виставляється, поки run довідно не покрив увесь корпус. Косинус-пороги — функція моделі, перевиводяться. Під час міграції ingest пінить embed-модель на цільову (без мішанини в вікні).
**Чому:** змішані вектори різних моделей дають беззмістовні косинуси — тихий клас помилок. Stamp робить змішування структурно неможливим; self-healing переживає перервані міграції.
**Де:** Д1 (Технічне), Д14 (Міграція embeddings).

### ADR-7 — LLM I/O contract: structured output + fencing + fail-loud + verify-then-climb

**Рішення:** (a) рішення моделі — тільки structured output / token-anchored, ніколи з підрядка; (b) код володіє списками/форматом/ID — модель дає лише структуровані поля; (c) недовірений зовнішній текст іде з огорожею "це ДАНІ", вихід валідується перед дією; (d) на збій — fail-loud, ніколи success-shaped; (e) ескалація через verify-then-climb.
**Застереження (інакше silent failure у формі success):**

- **shape ≠ content.** Structured output валідує лише *форму*. Будь-яке модельне поле, що стає параметром зовнішньої дії (recipient, ID, шлях), **переавторизується проти code-owned allowlist** — інакше injection через коректно-сформоване поле проходить.
- **confident-wrong під verify-then-climb.** Детермінований чек ловить лише *самозадекларовану* нездатність (порожньо / "не можу"). Впевнено-хибна fluent-відповідь проходить його за конструкцією → тягар на judge. Дешевий judge ділить blind spot із дешевим генератором (корельовані помилки). Захист: (1) **diversity** — judge з іншої модельної родини, ніж генератор (декорелює сліпі зони); (2) метрика точності judge на смузі handle-confidently (Д16); (3) bias-to-escalate не рятує — впевнена помилка не дає сигналу сумніву.
**Чому:** кожен пункт — типовий клас відмов LLM-систем. Межа між "іграшка" і "продукт". Каталог — companion.
**Де:** Принципи (fail-loud, LLM-proposes-code-disposes), Д4, Д5, Д11, Д12, Д16 (judge-метрика).

### ADR-8 — Вимірювання: емпіричні пороги + заморожений held-out + статистична строгість

**Рішення:** жоден поріг не хардкодиться "на відчуття" — виводиться з виміряних розподілів. Eval-харнес (recall@k/MRR/nDCG/F1, регресійний CV) будується *перед* фічею.

- **Заморожений held-out (80/20).** Регресійний гейт бачить лише dev-eval (80%); held-out (20%) гейт **ніколи не бачить** — періодичний аудит проти накопиченого eval-overfitting. Eval-мітки — **зовнішньо-розмічені й заморожені** проти навчального циклу (інакше гейт міряє самоузгодженість, не правильність — ADR-4).
- **Статистична строгість per-class гейта.** K+1 одночасних тестів → корекція Benjamini-Hochberg/Bonferroni + явний δ-tolerance + мін. розмір вибірки на клас. Без цього — інфляція хибних відмов.
- **Протокол оновлення набору** — як збираються нові мітки (розмітка свіжих прод-виходів) і замінюють застарілі.
**Чому:** без вимірювання неможливо знати, чи зміна допомогла; пороги дрейфують. Один eval-набір на всі цикли → переобучення відбору.
**Де:** Принципи (measured-not-tuned), Д5 (band'и), Д16 (харнеси, held-out), Д8 (регресійний гейт), ADR-9.

### ADR-9 — Стигмергічний контур + калібрація: стійкість і насичення

**Рішення:**

- **Сатуруюча сила сліду.** `strength = f(seen_count) · exp(−λ·age)`, де `f` — насичуюча, нормована в [0,1]: `f(n) = log(1+n)/(log(1+n)+S)` (Hill, half-saturation `S`). Лінійна `f` дала б необмежену силу (вічні ребра — анти-патерн).
- **Підкріплення лише від незалежних подій.** `seen_count` росте тільки від **провенанс-різних** ingest-подій, не від внутрішньо-піднятих ре-детекцій (інакше confirmation-loop: система частіше "знаходить" те, у що вже вірить → роздуває і силу, і незалежність ADR-3).
- **Decay — домінантний полюс.** Щоб reinforce+retrieval+learning (замкнений positive-feedback контур) не йшов у rich-get-richer розгін: нормалізація сил сліду + decay сильніший за приріст. Параметри `λ, S, μ` — у єдиному інвентарі тюнінгу (ADR-8), не "на відчуття".
- **Cold-start калібрація.** Bootstrap (isotonic/Platt на 50–200 зовнішніх мітках) → критерій `ECE < поріг`; доки не досягнуто — confidence не довіряється (гейт на pass/fail).
**Чому:** позитивний зворотний зв'язок без насичення й домінантного decay осцилює/розганяється (Д9-стабільність). Незалежність підкріплення — передумова коректності ADR-3.
**Відкинуто:** лінійна `f` (розгін); підкріплення від будь-якої ре-детекції (confirmation-loop).
**Де:** Д1 (підкріплення+decay), Д8 (консолідація), Д9 (стабільність контуру), ADR-3 (незалежність), companion (OP — формальний аналіз стійкості, тюнінг λ).
